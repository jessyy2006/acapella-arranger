# 001 — Soprano diverges from melody; arrangement MIDI missing Lead track

- **Type**: bug + enhancement (bundled)
- **Priority**: high (blocks Phase 5 demo quality)
- **Effort**: medium
- **Found in**: Phase 4 smoke test on `outputs/smoke_input.mp3`
- **Commit at time of report**: `d808b7b`

## TL;DR

Soprano (S) generated by the hybrid model doesn't track the input melody — notes are off, with silence gaps. Also, the output MIDI only has S/A/T/B; we want a 5th "Lead" part so the listener can compare what the audio pipeline extracted vs. what the model generated.

## Current behaviour

Smoke run (logged at `outputs/.smoke_run.log`):

- Demucs on MPS ~21s, CREPE-full ~22s, decode 4-voice ~16s
- Extracted 530 lead tokens
- S=140 / A=121 / T=84 / B=257 tokens
- Output: 4-part MIDI at `outputs/smoke_arrangement.mid`

Listening: S drifts off the input melody, has rests where the mp3 has sustained notes, hits wrong pitches.

## Expected behaviour

1. S should closely track the input melody while still harmonising (the 4 voices collectively form a chorale).
2. Output MIDI should have **5 parts** — Lead (what the audio pipeline extracted), S, A, T, B — so the listener can A/B the extraction against the generated soprano.

## Candidate root causes (to verify)

Source references in `src/pipeline/audio_to_midi.py` and `src/pipeline/run_pipeline.py`:

- **Silence gaps** — `confidence_threshold=0.5` at `audio_to_midi.py:173`: frames below that become REST (line 197). Demucs can thin the vocal stem in quiet sections; periodicity dips and the whole region drops to REST. Candidate fix: try 0.3–0.4, or add a hysteresis (raise/lower thresholds for enter/exit).
- **Notes off** — CREPE octave errors (documented as a known limitation at `audio_to_midi.py:25-26`). Median filter (`kernel_size=5` at line 192) smooths within-octave jitter but doesn't correct octave jumps. Candidate fix: add Viterbi post-processing (spec §"optional") or a simple same-direction octave-jump detector.
- **Min note filter** — `min_sixteenths=1` at line 227 drops any run shorter than a 16th. On fast passages this deletes real notes. Candidate fix: keep the filter but drop only if confidence < some higher bar.
- **S not attending to lead** — cross-attention in `src/models/hybrid.py:101-107` assumes the encoded lead memory carries pitch info. If the lead tokens are octave-shifted or REST-heavy, the memory is garbled and S falls back to prior harmony. This is downstream of the audio-pipeline issues above — fix those first, then re-listen.

## Acceptance criteria

1. `outputs/smoke_arrangement.mid` opens in MuseScore with 5 labelled parts (Lead, S, A, T, B).
2. Listening to the Lead track alone matches the input mp3 melody within tolerable CREPE accuracy (small octave/rhythm errors OK; large silence gaps or wrong-by-a-fourth errors not OK).
3. Listening to Lead + S together: S is clearly derived from the Lead (same melodic contour, harmonised), not a random walk.

## Fix plan (proposed order)

1. **Add Lead to output MIDI** (small — `run_pipeline.py` already has `lead_tokens` and `decode_part`; just insert a 5th part labelled "LEAD"). Re-run smoke test.
2. **Listen to the Lead part in isolation.** If Lead sounds wrong → audio-pipeline bug; go to (3). If Lead sounds right → model bug; go to (4).
3. **Audio pipeline tuning** — lower `confidence_threshold`, add Viterbi or octave-jump correction, re-test on the same smoke mp3.
4. **Model-side** — log cross-attention weights during decode; confirm S actually attends to the Lead memory. If not, the training data split may not have enforced S≈Lead tightly enough; revisit (out of Phase 4 scope).

## Risks / notes

- Don't regress the 148-green test suite. The Lead-track addition needs a new assertion in `tests/test_audio_pipeline.py` or in a new `tests/test_run_pipeline.py`.
- Audio-pipeline tuning can trade silence-gaps against spurious-notes — pick the axis that matters for the demo and note the tradeoff in the ticket on close.
- Viterbi post-processing is listed as optional in `docs/specs/audio_pipeline.md`; if we add it, commit message should cite the spec so the grader sees it.
